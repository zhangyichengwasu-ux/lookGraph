## 二、性能数据对比

### 2.1 会话统计总览

| 指标 | 第1轮（非LG） | 第2轮（LG初用） | 第3轮（LG熟练） | 非LG→LG 变化 |
|------|:-----------:|:------------:|:------------:|:----------:|
| **交互轮次** | 99 | 74 | 53 | **↓46%** |
| **工具调用** | 61 | 46 | 31 | **↓49%** |
| **LG 调用** | 0 | 42 | 26 | — |
| **Grep/Read** | 29 | 2 | 0 | **↓100%** |
| **输入 Token** | 405.4K | 420.9K | 470.1K | ↑16% |
| **输出 Token** | 49.7K | 65.9K | 51.1K | ~持平 |
| **总计 Token** | 455.1K | 486.9K | 521.3K | ↑15% |

### 2.2 核心指标解读

**交互效率大幅提升**：轮次从 99 降至 53，减少 46%；工具调用从 61 降至 31，减少 49%。原因是 LookGraph 一次 `semantic_search` + `class_context` 调用能同时返回代码结构、依赖关系、业务注释，而非 LG 模式下需要多个 Grep/Read 逐步拼凑信息。

**传统搜索完全被替代**：Grep/Read 从 29 次降至 0 次，说明 LookGraph 在"搜索代码"和"理解代码"场景下完全替代了传统工具。

**Token 消耗略增（反直觉但合理）**：总计 Token +15%，原因是 LG API 返回的信息密度远高于 Grep 匹配到的几行片段——`class_context.py` 一次性返回类源码、字段、方法签名、依赖关系、业务注释等完整上下文。本质上是**用 15% 的 Token 溢价换 46% 的轮次缩减**。

**熟练度效应明显**：同样是 LG 模式，R3 比 R2 少 21 轮次（74→53），工具调用少 15 次（46→31），说明熟悉工具链后有进一步优化空间。

**单轮效率换算**：非 LG 每轮承载约 1.0% 的任务进度，LG 每轮承载约 1.9%，**单轮效率提升约 87%**。

## 四、质量-效率 综合分析

### 4.1 核心发现

**LG 没有损害质量，但改变了质量的"形状"**：

- **无 LG 模式**：深度优先、事无巨细。通过 99 轮逐文件深挖，获得最深的字段级精度和最全的前置业务链路，但最终图的信息密度已超出 PlantUML 对象图的最佳承载范围。
- **LG 熟练模式**：广度优先、全局视角更好。用 53 轮（省 46%）达到 R1 约 85% 的完整度，在跨模块视角（支付→结算闭环）和可读性上反超 R1。
- **LG 初学模式**：产出质量与效率均未达最优，说明 LG 有明显的熟练度效应。

### 4.2 LG 模式的优势场景

| 场景 | 效果 |
|------|------|
| 快速建立全局认知 | ✅ 一次 project_summary 即可概览全貌 |
| 跨模块关联分析 | ✅ 语义搜索可跨包/跨模块定位 |
| 理解已有业务注释 | ✅ class_context 直接返回历史注释 |
| 探查未知业务领域 | ✅ semantic_search 用自然语言定位代码 |
| 字段级精确分析 | ⚠️ 仍需 Read 补充，但定位速度已提速 |

## 五、结论与建议

### 5.1 总结

| 维度 | 结论 |
|------|------|
| **交互效率** | ✅ 大幅提升，轮次减半，单轮效率提升 87% |
| **代码定位** | ✅ Grep/Read 完全被替代 |
| **Token 成本** | ⚠️ 略增 15%，但换来了更丰富的信息密度和更少的交互轮次 |
| **产出质量** | ✅ 与无 LG 持平或更优（在不同维度各有胜负） |
| **学习效应** | ✅ 熟练度越高，效率和质量的综合收益越明显 |

### 5.2 建议

1. **日常代码分析优先使用 LookGraph**：在大多数场景下，LG 能用更少的轮次产出同等或更优质量的结果。
2. **字段级精确分析时辅以 Read**：当需要确认具体字段名、类型、注释时，在 LG 定位到目标类后，用 Read 补充确认。
3. **善用语义注释形成正向循环**：项目中积累的业务注释越多，后续 LG 的搜索和理解效果越好。
4. **初用 LG 时预留学习曲线**：R2→R3 的提升说明，经过 1-2 次任务熟悉后，LG 的效能才能完全释放。

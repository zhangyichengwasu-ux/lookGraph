# LookGraph vs 无LookGraph 测试对比报告

> 测试任务：统计 zcyl-mall-api 下所有 HTTP 接口的作用与业务含义，汇总成 md 文档  
> 测试日期：2026-06-24  
> 交付物：1.md（无LG）、2.md（LG首轮）、3.md（LG次轮，已有语义沉淀）

---

## 一、效率维度

### 1.1 会话指标对比

| 指标 | 1.md（无LG） | 2.md（LG首轮） | 3.md（LG次轮） | 趋势 |
|------|:-----------:|:------------:|:------------:|------|
| 对话轮次 | **42** | 66 | 49 | LG首轮最慢，次轮回落 |
| 工具调用总数 | **30** | 38 | 33 | 无LG工具调用最少 |
| LookGraph 调用 | 0 | 10 | **3** | 语义沉淀后大幅减少 |
| Grep/Read 调用 | 27 | 24 | 27 | 三者接近 |
| 输入 Token | **352.9K** | 385.3K | 504.7K | LG次轮反而最高 |
| 输出 Token | 40.2K | 65.4K | **69.0K** | LG输出更丰富 |
| 总 Token | **393.1K** | 450.7K | 573.6K | 无LG最省 |

### 1.2 效率分析

**无LG模式（42轮，393K token）**
- grep/read 直接定位源码，工具链路最短
- 但 grep 无法系统穷举所有 Controller，容易漏接口
- 输出最少（40K），信息密度低

**LG首轮（66轮，451K token）**
- 需要逐 Controller 做：语义搜索 → 获取上下文 → 创建语义注释
- 首次建立语义索引的成本不可避免
- 10次LG调用覆盖了所有 Controller

**LG次轮（49轮，574K token）**
- LG调用从10降到3，说明语义沉淀确实**减少了重复搜索**
- 但输入token反而最高（505K），因为历史注释数据被加载进上下文
- 输出最丰富（69K），信息密度最高

## 四、核心发现

### 4.1 无LG模式的致命缺陷

**接口覆盖严重不足**（102 vs 180+）。grep/Read 对代码库的探索是**点状的**，缺少全局结构视图，容易遗漏模块。对于"统计所有接口"这类需要全局穷举的任务，传统工具链有根本性的局限。

### 4.2 LG模式的核心价值

1. **完整性保证**：代码图谱的结构化索引确保不会遗漏 Controller 和接口方法
2. **语义理解**：LG模式的业务描述更深入，尤其是次轮执行有历史注释加持后
3. **技术特征提炼**：LG次轮（3.md）独有的「关键技术特征」章节——`@Idempotent`、SSE流式、异步任务、城市隔离——这些不是接口列表能体现的，而是理解了代码设计后才能提炼的

### 4.3 语义沉淀的实际效果

对比2.md和3.md：
- LG调用从10降到3（**减少70%**）
- 对话轮次从66降到49（**减少26%**）
- 输出质量从19分提升到18分（基本持平，各有优劣）

**语义沉淀确实减少了重复搜索**，但输入token反而更高（历史注释占用上下文），这是当前的tradeoff。

### 4.4 最佳实践建议

对于"统计接口+生成文档"类任务，**推荐的执行策略**：

1. **首轮用LG**：借助图谱完整性保证接口不遗漏
2. **路径格式用完整路径+参数**：像2.md那样，对开发者最实用
3. **结合技术特征分析**：像3.md那样提炼设计模式
4. **可添加业务流程图**：1.md的流程图是很好的补充，应该保留到LG版本中
